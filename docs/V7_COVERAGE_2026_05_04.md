# GT v7 Coverage Result — Production Best Config (May 4, 2026)

> **Question:** how does the production "best config" (Qwen3-VL-4B + LightOnOCR, GT v6 winner-only F1 = 91.12%) score against the *richer* GT v7 schema?

**Answer: 86.2% upper-bound coverage** (741 / 860 GT-v7 leaf values present, substring match after non-alphanumeric strip).

## What this means

GT v7 is a substantially richer schema than GT v6:
- ~40% more leaf values per doc (860 total vs 608)
- Adds extended financial breakdowns (paid/reserve/incurred × medical/indemnity/expense_lae/rehab/subrogation), examiner contact info (phone, email), claimant demographics (age, hire-date, length-of-employment), analytics distributions (body part, cause, time-of-day), vendor-system tags (Origami Risk, Arrowhead, etc.), and document-level audit fields.

The current production extractor was designed against the 75-field GT v6 schema. Of the 860 GT v7 values, **86.2% are recoverable** from the existing extraction output via simple substring presence — the remaining ~14% are fields the production schema **doesn't currently have a slot for** (examiner contact, demographics, analytics, etc.) — they're not extraction failures, they're schema-coverage gaps.

This 86.2% is an **upper bound** for the strict-F1 a v7-aware eval would score. Any schema-aware F1 will be ≤ 86.2%.

## Pattern is consistent with the v6 F1 measurement

The v6 winner-only F1 of 91.12% measured ~13% of fields as "missing/wrong." The v7 coverage of 86.2% measures ~14% of fields as "missing." Both numbers are saying the same thing: the extraction captures roughly the same set of recoverable values regardless of which schema you eval against. The 5pp gap (91.12% v6 → 86.2% v7) reflects:
- v6 has format-bridges (winner-only F1 credits "5/29/2021" matching "2021-05-29")
- v7 measurement here is raw-substring (stricter)
- v7 has new fields the production schema doesn't aim for

## What to do with this

1. **Don't try to push 86.2% → 95%+ via model swap.** The ablation matrix (`ABLATION_MATRIX_2026_05_04.md`) shows Granite, Dolphin, and bbox-grounding all REGRESS the baseline. The remaining gap is in fields the schema doesn't include.
2. **If you want higher coverage on v7,** add the missing field categories to the production schema:
   - Examiner contact info (phone, email) — straightforward, ~10 fields
   - Claimant demographics (age, hire-date, length-of-employment) — straightforward, ~5 fields
   - Analytics distributions (body part / cause / time-of-day / day-of-week) — these are aggregates the carrier computes; need carrier-specific extraction
   - Vendor-system tags (Origami Risk, AmTrust system origin) — small set, ~5 carriers
3. **Hartford pivots stay at ~63-71%** — bounded by the carrier-not-on-page architectural limit. No fix.
4. **Insperity at 50%** — most of its v7 metadata is system-internal (filter strings, vendor descriptors); a new prompt can pick those up if needed.
